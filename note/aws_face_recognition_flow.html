Two journeys

Register กับ Verify ไม่ใช่ API เดียวกัน

01

User login แล้วกด Register face

UI เช็กว่า user มี session ของ app แล้ว และ faceEnrollmentStatus ยังไม่ใช่ registered จึงแสดง CTA แบบไม่รบกวน navbar

ไม่ call AWS
แค่ render จากสถานะ user ใน DB
02

Backend สร้าง Face Liveness session

เมื่อ user กดเริ่มจริง ๆ UI ค่อยเรียก POST /api/face/session แล้ว backend เรียก CreateFaceLivenessSession

BE → Rekognition ใช้ AWS profile local หรือ IAM role บน production
03

Frontend เปิดกล้องและ stream liveness

ตัว detector ใช้ Cognito Identity Pool เพื่อแลก Google ID token เป็น temporary AWS credential สำหรับ StartFaceLivenessSession เท่านั้น

FE → Cognito → Rekognition credential สั้นและ scope แคบ ไม่ใช่ access key ถาวร
04

Backend อ่านผล liveness หนึ่งครั้งต่อ request

GET /api/face/result เรียก GetFaceLivenessSessionResults แล้วเก็บ referenceS3Key ถ้า outcome accepted

BE → Rekognition ไม่ poll รัว ๆ; terminal result ถูก cache ใน DB
05

IndexFaces เพิ่มใบหน้าเข้า collection

Backend ใช้ reference image จาก S3 แล้วเรียก IndexFaces กับ atk-store-face-collection-dev โดยส่ง ExternalImageId เป็น stable app identifier

BE → S3 → Rekognition ควรใช้ QualityFilter และตรวจว่ามีหน้าเดียว
06

DB เก็บ FaceId mapping

เมื่อ AWS ส่ง FaceId และ ImageId กลับมา backend ค่อยอัปเดต DB ว่า user คนนี้ register face แล้ว

DB write
userId, collectionId, faceId, registeredAt
01

User ต้องผ่าน app auth ก่อน

Verify identity ไม่เริ่มจากกล้องเปล่า ๆ แต่เริ่มจาก session ของ app เพื่อรู้ว่า request นี้เป็นของใคร และป้องกัน anonymous probe

Auth gate
ใช้ atk_session ของระบบเรา
02

ทำ Face Liveness ใหม่

ทุกครั้งที่ verify สำคัญ ควรมี liveness ใหม่ก่อน search เพื่อให้รูปที่จะเทียบเป็นภาพจากมนุษย์จริง ณ session นั้น

Create → Start → Get flow เดิม แต่ intent คือ verify ไม่ใช่ enrollment
03

SearchFacesByImage ใน collection

Backend ส่ง reference image ที่ผ่าน liveness เข้า SearchFacesByImage พร้อม threshold สูง เช่น 99 สำหรับ identity verification

BE → Rekognition ผลลัพธ์คือ candidate faces และ similarity score
04

Backend map FaceId กลับเป็น user

ถ้า match ที่ดีที่สุดมี FaceId อยู่ใน DB และ similarity ผ่าน threshold ระบบจึงรู้ว่าใบหน้านี้คือ user คนไหน

DB lookup
FaceId → userId
05

ตัดสินผลแบบ server-authoritative

ผลสุดท้ายไม่ควรให้ frontend ตัดสินเอง Backend ต้องรวมผล liveness, face match, ownership และ policy เช่น retry limit

Decision

API economy

กฎลดการ call API โดยไม่ลดความน่าเชื่อถือ

RULE 01

ไม่เปิดกล้องจาก page load

UI แสดง CTA ก่อนเสมอ และเริ่ม session เมื่อ user กดเท่านั้น ลดทั้งค่าใช้จ่ายและ false start

RULE 02

หนึ่ง intent หนึ่ง liveness session

Register หนึ่งครั้งใช้หนึ่ง session; verify หนึ่งครั้งใช้หนึ่ง session ไม่สร้างใหม่ระหว่างยัง pending

RULE 03

IndexFaces เฉพาะหลัง accepted

ถ้า liveness ไม่ผ่าน ห้ามเอารูปไป index เพื่อลด noise ใน collection และลด API ที่ไม่จำเป็น

RULE 04

SearchFacesByImage เฉพาะจุดเสี่ยง

ไม่ search ทุกหน้า ใช้เมื่อ register กัน duplicate หรือเมื่อ flow ต้อง verify identity จริง

เหตุผลที่ต้องเอา liveness มาก่อน recognition: face recognition ตอบว่า “รูปนี้คล้ายใครใน collection” แต่ไม่ได้ตอบว่ารูปนั้นมาจากคนจริงหน้ากล้องในเวลานั้นหรือไม่ ดังนั้น flow ที่ปลอดภัยกว่าคือ liveness accepted ก่อน แล้วค่อย IndexFaces หรือ SearchFacesByImage
Backend decision

จุดที่ backend ต้องตัดสิน

Frontend แสดงผลได้ แต่ไม่ควรเป็นเจ้าของความจริงของ identity เพราะผลจาก AWS ต้องถูกรวมกับ session ownership และ DB policy

Accepted

liveness score ผ่าน threshold และ face match similarity ผ่านค่า เช่น AWS_FACE_MATCH_THRESHOLD=99

server authority รวมผลจาก AWS + DB + app session

Rejected or review

liveness failed, no face found, multiple faces, similarity ต่ำ, FaceId ไม่อยู่ใน DB, หรือ match เป็น user คนอื่น
