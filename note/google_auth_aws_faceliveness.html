ATK Store Identity Flow

Google login เปิดประตู, AWS ตรวจว่าคนจริง

ภาพนี้อธิบายว่า Google Authentication, Application Session, Cognito Identity Pool, AWS Profile และ Rekognition Face Liveness ทำงานร่วมกันอย่างไร โดยแยกให้ชัดว่าอะไรอยู่ใน browser, อะไรอยู่บน server และ token ไหนใช้เพื่ออะไร

Normal liveness attempt Create ×1, Start ×1, Get ×1
Browser sees sessionId + temporary AWS creds
Browser never sees AWS profile, IAM secret, Google secret
Region anchor ap-northeast-1

ตัวละครหลัก 4 ฝั่ง

ถ้าจำแค่นี้ก่อนก็พอ: Google บอกว่า user คือใคร, App ออก session ของเราเอง, Cognito แปลง Google identity เป็น AWS temporary credentials, Rekognition ทำ liveness จริง

G

Google Authentication

ยืนยันตัวตนผู้ใช้ผ่าน OAuth/OIDC แล้วส่ง id_token กลับมาให้ backend ตรวจ signature และ claims

APP

Next.js App Session

หลังตรวจ Google ผ่าน server จะสร้าง atk_session ของแอปเอง ใช้คุมว่า request นี้เป็น user คนไหน

CI

Cognito Identity Pool

i

Face Liveness ฝั่ง browser ต้อง call StartFaceLivenessSession ไปที่ AWS โดยตรง แต่ user ของเรา login ด้วย Google ไม่ได้มี AWS account หรือ IAM key เป็นของตัวเอง

Cognito Identity Pool จึงเป็นสะพานที่ AWS ไว้ใจ: เอา Google ID token ที่ผ่านการยืนยันแล้ว มาแลกเป็น temporary AWS credentials อายุสั้นและจำกัดสิทธิ์เฉพาะงานเปิดกล้อง liveness

เป็นสะพานแลก Google ID token เป็น AWS temporary credentials ที่ scope ไว้ให้เริ่ม Face Liveness เท่านั้น

AWS

AWS Profile + Rekognition

flow

การเปิดกล้องไม่ได้เกิดตอนเข้าเว็บ แต่เกิดเมื่อ user ที่ login แล้วกดเริ่ม และระบบมี session/credential ครบเท่านั้น

  1. UI แสดง CTA ลงทะเบียนใบหน้า เงื่อนไข: user มี atk_session และ faceEnrollmentStatus ยังไม่ใช่ registered
  2. user กดเริ่ม → UI ขอสร้าง session POST /api/face/session; มี startingRef กันกดซ้ำ ไม่ call ตอน page mount
  3. server ตรวจ auth + origin แล้ว Create เงื่อนไข: same-origin, requireCurrentUser() ผ่าน, env AWS ครบ; เรียก CreateFaceLivenessSession แค่ครั้งเดียว หรือ reuse pending session ที่ยังไม่หมดอายุ
  4. server ส่ง sessionId กลับ UI UI เปลี่ยน phase เป็น detecting แล้วจึง mount FaceLivenessDetectorCore
  5. detector ขอ temporary credentials GET /api/face/credentials; server ใช้ Google ID token bridge ผ่าน Cognito Identity Pool
  6. เปิดกล้องและ Start Face Liveness เงื่อนไข: มี sessionId, region, และ temporary AWS credentials; browser จึง call StartFaceLivenessSession
  7. analysis complete → server อ่านผล POST /api/face/result; server call GetFaceLivenessSessionResults หนึ่งครั้ง และ retry ได้อีกหนึ่งครั้งเฉพาะผลยัง pending

กฎ call API: normal attempt = Create ×1 + Start ×1 + Get ×1; ไม่มี polling และไม่มีการเปิดกล้องอัตโนมัติ

ฝั่ง backend ใช้ AWS Profile สำหรับ Create/Get session; ฝั่ง browser ใช้ temporary credentials สำหรับ Start เท่านั้น

Flow ตั้งแต่ login จนเปิดกล้อง

เส้นนี้คือของจริงในระบบปัจจุบัน: login ก่อน, มี session ของแอปก่อน, แล้วค่อยกดเริ่มลงทะเบียนใบหน้า จึงจะเกิด AWS call

01
User กดเข้าสู่ระบบด้วย Google ไปที่ /api/auth/signin/google
App สร้าง state, PKCE, nonce เก็บใน httpOnly cookie สั้น ๆ
02
Google callback กลับมาที่ backend server แลก code เป็น token แล้วตรวจ id_token
03
App ออก session ของตัวเอง ตั้ง atk_session และเก็บ Google ID token แบบ path-scoped ไว้ที่ /api/face
04
User เห็น prompt ให้ register ใบหน้า ยังไม่เปิดกล้อง และยังไม่ call AWS
05
User กดเริ่ม browser POST ไป /api/face/session
Backend ใช้ AWS Profile เรียก Rekognition CreateFaceLivenessSession แล้วส่ง sessionId กลับ
Rekognition สร้าง liveness session single-use, อายุสั้น ประมาณ 3 นาที

Rekognition วิเคราะห์ liveness อาจบันทึก reference/audit image ไป S3 Tokyo
08
Analysis complete browser แจ้ง backend ที่ /api/face/result
Backend ตัดสินผล เรียก GetFaceLivenessSessionResults หนึ่งครั้ง, เทียบ threshold, update user flag
ได้ confidence + reference image ยังไม่ใช่ face vector และยังไม่ใช่ face recognition

Application session

atk_session

  • บอกว่า user login อยู่กับแอปเรา
  • ใช้กับทุก protected page/API
  • server ตรวจจาก DB session hash
Liveness session

sessionId

  • สร้างโดย Rekognition
  • ใช้ครั้งเดียวกับการเปิดกล้องหนึ่ง attempt
  • หมดอายุสั้น ประมาณ 3 นาที
AWS credential session

temporary keys

  • ออกโดย Cognito Identity Pool
  • มี access key, secret, session token แบบชั่วคราว
  • จำกัดสิทธิ์ไว้สำหรับ detector

API call budget: ทำยังไงไม่ให้เรียกรัว

เป้าคือหนึ่ง attempt ที่ดีควรมี AWS liveness call เท่าที่จำเป็น และไม่เริ่มเองจนกว่า user จะกด CTA

1

CreateFaceLivenessSession

server เรียกเมื่อ user กดเริ่มเท่านั้น ถ้ามี pending attempt ที่ยังไม่หมดอายุ จะ reuse session เดิมแทนการสร้างใหม่

1

StartFaceLivenessSession

browser detector เรียกเองตอนเปิดกล้อง โดยใช้ sessionId และ temporary credentials จาก Cognito

1

GetFaceLivenessSessionResults

server เรียกหลัง detector แจ้งว่า analysis complete เท่านั้น ปกติไม่ poll; ถ้า pending อนุญาต retry หน่วงเวลาได้อีกหนึ่งครั้ง

AWS Profile อยู่ตรงไหน

AWS Profile เป็นของ backend/local developer เท่านั้น ไม่ใช่ของ browser และไม่ใช่ของ customer ที่ login Google

Backend side

  • local ใช้ AWS_PROFILE=atk-store-dev ผ่าน AWS SDK default provider chain
  • ใช้เรียก CreateFaceLivenessSession และ GetFaceLivenessSessionResults
  • production ควรเปลี่ยนเป็น IAM role ของ server/VPS ที่จำกัดสิทธิ์แทน profile file

Browser side

  • ไม่ควรมี IAM access key หรือ secret access key ถาวร
  • ได้ temporary credentials เฉพาะตอนจะเปิดกล้อง liveness
  • credentials ควรมีสิทธิ์เฉพาะ rekognition:StartFaceLivenessSession
สิ่งที่ใช้ ใครถือ เอาไว้ทำอะไร ส่งให้ browser ไหม
GOOGLE_CLIENT_SECRET backend แลก OAuth code เป็น token ไม่ส่ง
AWS_PROFILE backend local machine ให้ AWS SDK หา credential สำหรับ Create/Get ไม่ส่ง
atk_session browser cookie + DB hash session ของแอปเรา เป็น httpOnly cookie
atk_g_idt browser cookie path /api/face ให้ backend แลก Cognito credentials ส่งเฉพาะ request ไป /api/face, JS อ่านไม่ได้
temporary AWS credentials browser detector เรียก StartFaceLivenessSession ส่งเฉพาะแบบอายุสั้นและ scope แคบ

ได้อะไรจาก liveness และยังไม่ได้อะไร

ขั้นนี้พิสูจน์ว่า “คนจริงอยู่หน้ากล้อง” ยังไม่ใช่การจำแนกว่า “คนนี้คือนาย A”

ได้จาก Face Liveness

  • status เช่น succeeded, failed, expired
  • confidence score เพื่อเทียบ threshold
  • reference image / audit image ใน S3 ถ้าตั้งค่า output
  • flag ใน user table ว่า registered หรือยัง

ยังไม่ได้ในรอบนี้

  • ยังไม่สร้าง face vector เองในระบบเรา
  • ยังไม่สร้าง Rekognition Face Collection
  • ยังไม่ทำ IndexFaces หรือ SearchFacesByImage
  • ยังไม่รู้ว่าเป็น “นาย A” จนกว่าจะเพิ่ม recognition phase
